Nombre:           Zero-Knowledge Vault
Tipo:             Gestor de contraseñas Zero-Knowledge para equipos
Licencia:         MIT

Lenguaje:         TypeScript / React / Next.js
Runtime:          Node.js (Bun 1.3+ compatible)
Base de datos:    SQLite (dev) / PostgreSQL (prod)
Cache:            Redis (opcional)

Descripción del producto:
  Gestor de contraseñas donde TODO el cifrado ocurre en el navegador del
  cliente usando Web Crypto API. El servidor es un "crypto-blind store"
  que solo almacena blobs cifrados, sales públicas, IVs y llaves públicas.
  NUNCA recibe contraseñas maestras, llaves privadas en claro, ni el
  contenido de ningún secreto.

 5. MODELOS DE DATOS PRISMA (6 modelos)
================================================================================

1. User
   - id (String, @id, cuid)
   - email (String, @unique)
   - name (String?)
   - createdAt, updatedAt (DateTime)
   - Relaciones: keyMaterial, ownedSecrets, secretKeyShares, devices, auditLogs

2. UserKeyMaterial
   - id, userId (@unique)
   - kdfAlgorithm (default "argon2id")
   - kdfSalt, kdfIterations, kdfMemoryKiB?, kdfParallelism?
   - publicKeyJwk, publicKeyFingerprint
   - popSignature, popSignatureHash
   - encryptedPrivateKeyJwk, privateKeyIv
   - mlKemPublicKey?, encryptedMlKemPrivateKey?, mlKemPrivateKeyIv?
   - recoverySalt?, recoveryIterations?, encryptedPrivateKeyForRecovery?
   - recoveryIv?, recoveryEnabled (Boolean)
   - createdAt, updatedAt

3. Secret
   - id, ownerId
   - encryptedTitle, titleIv (AES-256-GCM ciphertext)
   - encryptedData, dataIv (AES-256-GCM ciphertext)
   - createdAt, updatedAt
   - Relación: keyShares[]

4. SecretKeyShare
   - id, secretId, recipientId
   - wrappedSymmetricKey (RSA-OAEP wrapped AES key)
   - role (default "readonly")
   - seenAt? (DateTime?)
   - createdAt
   - @@unique([secretId, recipientId])

5. Device
   - id, userId, deviceName
   - publicKeyECDH (P-256, JWK JSON)
   - publicKeyECDHFingerprint (SHA-256 hex)
   - enrollerPublicKeyECDH? (JWK del dispositivo autorizador)
   - wrappedPrivateKeyForDevice, wrappedPrivateKeyIv (AES-256-GCM)
   - enrollCode? (6 dígitos)
   - enrollCodeExpiresAt?
   - enrolledAt, lastSeenAt
   - revokedAt?

6. AuditLog
   - id, userId
   - encryptedEvent, eventIv (AES-256-GCM ciphertext)
   - encryptedCategory, categoryIv
   - eventCategory (String: auth|secret|share|device|recovery)
   - prevHash? (SHA-256 del log anterior)
   - logHash? (SHA-256 del log actual)
   - createdAt
   - @@index([userId, createdAt])
   - @@index([eventCategory])

 6. STACK CRIPTOGRÁFICO (implementado y testeado)
================================================================================

KDF primario:        Argon2id (m=64MiB, t=3, p=4) vía hash-wasm en Web Worker
KDF fallback:        PBKDF2-SHA256 (600,000 iteraciones)
Subkey derivation:   HKDF-SHA256 (RFC 5869) — 4 propósitos: audit, device, share, metadata
Cifrado simétrico:   AES-256-GCM (96-bit IV aleatorio, AAD opcional)
Wrap asimétrico:     RSA-OAEP 2048-bit (SHA-256)
Firmas:              RSA-PSS 2048-bit (salt=32, SHA-256) — Proof-of-Possession
Multi-device:        ECDH P-256 (Curve25519 equivalente NIST)
Device enrollment:   ECDSA P-256 (SHA-256) — challenge-response
Post-cuántico:       ML-KEM-768 (NIST FIPS 203) — hybrid con ECDH, activo en share flow
Recovery:            BIP-39 (24 palabras, 256 bits de entropía)
Audit chain:         SHA-256 hash chain (tamper-evident, verificable vía API)

Cumplimiento estándares:
  - Web Crypto API (W3C)
  - RFC 5869 (HKDF)
  - RFC 8017 (RSA-OAEP/PSS)
  - NIST FIPS 203 (ML-KEM-768)
  - BIP-39 (mnemonic recovery)
  - OWASP 2024 (Argon2id params)
